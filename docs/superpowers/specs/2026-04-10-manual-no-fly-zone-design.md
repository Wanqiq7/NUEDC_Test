# 手动设置禁飞区设计方案

## 背景

当前地面站已经具备以下能力：

- 通过 `ZeroMQ + Protobuf` 接收 Python 模拟端下发的 `GridConfig`
- 在 `GridScene` 中显示禁飞区、起飞区、终点、降落走廊与整条巡查航线
- 使用 Python 端 `plan_route(...)` 基于 `no_fly_cells` 生成覆盖航线

但现有系统仍缺少一个关键比赛链路：**在任务开始前，由地面站人工设置禁飞区，再基于新的禁飞区生成任务航线。**

根据赛题原文，本轮功能必须满足以下硬约束：

- 巡查前设置禁飞区
- 在地面站设置禁飞区
- 禁飞区为 **3 个连续方格**
- 本项目进一步收敛为：只允许 **横向 3 连格** 或 **纵向 3 连格**
- 生成后的航线必须覆盖禁飞区以外全部方格

## 目标

在不推翻现有 Qt Widgets 架构、不重写 Python 路径规划核心算法的前提下，新增一条完整的“人工禁飞区设置 → 合法性校验 → 航线生成 → 地图刷新 → 下发模拟端 / 无人机链路”的起飞前规划闭环。

本轮目标包括：

- 在界面中只保留 **一个主按钮**
- 主按钮初始文案为 `设置禁飞区`
- 点击后进入地图选格模式
- 用户在地图中点击 3 个格子形成合法直线禁飞区后，主按钮文案切换为 `航线生成`
- 点击 `航线生成` 后执行重规划并刷新显示
- 生成成功后同步更新：
  - 地面站本地地图预览
  - Python 模拟端使用的禁飞区与航线
  - 后续无人机执行链路所需的数据结构

## 已确认方案

采用 **A1：按钮驱动的显式状态机方案**。

### 选择原因

相较于“自动生成”或“纯隐式逻辑”方案，显式状态机更符合比赛演示与现场操作需求：

- 用户更容易理解当前处于“正在设置”还是“可以生成”
- 可以把非法输入、未选满 3 格、生成失败等情况清晰反馈出来
- 与现有 Qt 工程的事件驱动结构天然兼容
- 修改范围可控，不需要引入过度抽象的新架构层

## 交互设计

### 单按钮交互规则

系统仅保留一个主按钮，其行为随状态变化：

1. **初始状态**
   - 按钮文案：`设置禁飞区`
   - 地图处于只读预览状态
   - 若已有默认案例禁飞区，仅显示，不允许直接编辑

2. **进入设置态**
   - 用户点击 `设置禁飞区`
   - 地图进入“禁飞区选择模式”
   - 已有禁飞区显示被清空，进入新的候选选择流程
   - 按钮保持存在，但在未形成合法 3 格直线前不执行生成逻辑

3. **候选合法态**
   - 当用户在地图中选中的 3 个格子满足“横向三连”或“纵向三连”时
   - 按钮文案切换为：`航线生成`
   - 地图显示候选禁飞区高亮

4. **生成执行态**
   - 用户点击 `航线生成`
   - 系统执行禁飞区校验、航线重规划、地图刷新与链路同步
   - 成功后退出编辑态，回到只读预览
   - 按钮文案恢复为：`设置禁飞区`

### 地图交互规则

- 只有在“禁飞区选择模式”下，地图点击才生效
- 单击空白可选格子：加入候选集合
- 单击已选候选格子：取消选择
- 候选格子超过 3 个时，不允许继续新增，并给出状态提示
- 起飞格不允许被选为禁飞区
- 若未来存在固定降落终点约束，可扩展为终点格也不允许选中，但本轮先不强制

### 用户可感知反馈

- 候选态使用与正式禁飞区不同的颜色，避免用户误以为已经生成成功
- 状态栏实时提示当前进度，例如：
  - `已选择 1/3 个禁飞格`
  - `已选择 2/3 个禁飞格`
  - `已选择 3/3 个禁飞格，可生成航线`
  - `非法：3 个格子必须横向或纵向连续`
  - `航线生成成功，可执行任务`

## 状态机设计

建议在 `MainWindow` 内维护一个轻量显式状态枚举，例如：

- `IdlePreview`
- `SelectingNoFlyZone`
- `ReadyToGenerate`

### 状态转换

- `IdlePreview` --点击按钮--> `SelectingNoFlyZone`
- `SelectingNoFlyZone` --选出合法三格--> `ReadyToGenerate`
- `ReadyToGenerate` --取消某格或变为非法--> `SelectingNoFlyZone`
- `ReadyToGenerate` --点击按钮且生成成功--> `IdlePreview`
- `ReadyToGenerate` --点击按钮但生成失败--> 保持 `ReadyToGenerate`

### 设计原则

- 不把“按钮文案”当作状态来源，按钮文案只是状态表现
- 状态切换由数据条件驱动：候选格集合、合法性、规划结果
- 地图是否可点击，由状态统一控制，不由散落布尔变量控制

## 业务规则与合法性校验

### 基础合法性

- 必须恰好选择 3 个不同格子
- 3 个格子都必须在 9×7 地图范围内
- 不能包含起飞格

### 形状合法性

仅允许以下两类：

1. **横向三连**
   - 三个格子的 `B` 坐标相同
   - 三个格子的 `A` 坐标排序后连续

2. **纵向三连**
   - 三个格子的 `A` 坐标相同
   - 三个格子的 `B` 坐标排序后连续

### 明确禁止

- L 形三格
- 对角线三格
- 间隔跳格
- 超过 3 格
- 少于 3 格直接生成

## 架构与职责拆分

### `MainWindow`

负责：

- 创建并管理单按钮
- 维护规划 UI 状态机
- 响应地图选格信号
- 触发禁飞区合法性校验
- 在点击 `航线生成` 时调用规划链路
- 更新状态栏、任务说明、图例文案

建议新增的职责方法包括：

- 进入禁飞区设置模式
- 处理候选禁飞区变化
- 更新按钮文案与可用状态
- 触发航线生成
- 处理生成成功 / 失败后的 UI 回退

### `GridScene`

负责：

- 管理地图中每个格子的点击命中
- 提供“是否处于编辑模式”的开关
- 管理候选禁飞区与正式禁飞区两套显示
- 将用户点击的格子以信号形式回传给 `MainWindow`

建议新增能力：

- `setNoFlyEditEnabled(bool enabled)`
- `setCandidateNoFlyCells(const QStringList &cells)`
- `clearCandidateNoFlyCells()`
- 格子点击信号，例如 `cellClicked(QString cell_code)`

### Python 规划端

Python 端已具备 `plan_route(..., no_fly_cells=...)` 能力，因此本轮重点不是改算法，而是补一个**可复用的重规划入口**，让 Qt 地面站能够把新的禁飞区传进去并获得新的 `route`。

建议补一个小型服务接口或桥接函数，职责是：

- 接收最新 `start_cell`
- 接收最新 `no_fly_cells`
- 保持与当前 `mission_mode` / `landing_profile` 一致
- 返回新的 `route`、`terminal_cell` 与关联任务配置

### 通信与下发链路

当前仓库中已有 Python → Qt 的单向推送链路，但你要求“生成成功后同时更新并下发到模拟器 / 无人机链路”，因此本轮设计上应补一条**地面站主动发起的任务配置更新链路**。

推荐分两阶段落地：

1. **阶段一：本地闭环**
   - 地面站点击生成
   - 本地调用 Python 规划逻辑
   - 直接刷新本地地图与任务配置

2. **阶段二：对外下发**
   - 将新禁飞区与新航线编码为统一任务配置
   - 下发给模拟端 / 飞控接入层

如果你本轮只做仓库内联调，建议先完成阶段一，并把阶段二接口预留出来。

## 数据流设计

建议数据流为：

1. 用户点击 `设置禁飞区`
2. `MainWindow` 进入 `SelectingNoFlyZone`
3. `GridScene` 开始接收格子点击
4. 每次点击后把候选集合回传给 `MainWindow`
5. `MainWindow` 执行合法性校验
6. 若合法，状态切到 `ReadyToGenerate`，按钮文案变为 `航线生成`
7. 用户点击 `航线生成`
8. 地面站调用重规划入口，传入新的 `no_fly_cells`
9. 获得新的 `route / terminal_cell / 任务配置`
10. 更新 `GridScene` 的正式禁飞区与路线显示
11. 同步更新本地任务状态与后续下发配置
12. 返回 `IdlePreview`，等待执行任务

## 与现有代码的对接建议

### 现有可复用部分

- `GridConfig.no_fly_cells`：协议层已有字段，可直接复用
- `GridScene::setNoFlyCells(...)`：正式禁飞区显示能力已存在
- `GridScene::setRoute(...)`：新路线显示能力已存在
- `python/uav_testbed/route_planner.py`：已支持传入新的禁飞区重新生成路径

### 现有缺口

- `GridScene` 没有点击选格能力
- `MainWindow` 没有规划控制按钮与状态机
- Qt 侧没有主动触发 Python 重规划的桥接能力
- 当前链路是 Python 主动推送配置，缺少 Qt 主导的新任务生成路径

## 错误处理

### 输入错误

- 选择不足 3 格：不给生成
- 选择超过 3 格：拒绝新增并提示
- 三格不成直线：按钮不进入 `航线生成`
- 选择起飞格：立即提示并拒绝

### 规划错误

- 若新禁飞区导致规划失败：
  - 保留候选禁飞区显示
  - 不覆盖当前正式航线
  - 状态栏提示失败原因
  - 允许用户继续修改 3 个格子

### 链路同步错误

- 若本地已生成成功，但外部下发失败：
  - 明确区分“本地预览成功”与“任务下发失败”
  - 不应默默吞掉错误

## 测试策略

### Qt 侧建议新增测试

1. 三格横向连续时，校验通过
2. 三格纵向连续时，校验通过
3. L 形三格时，校验失败
4. 跳格时，校验失败
5. 包含起飞格时，校验失败
6. 进入设置态后按钮文案正确切换
7. 生成成功后按钮文案恢复为 `设置禁飞区`

### Python 侧建议新增测试

1. 新禁飞区输入后仍能覆盖所有非禁飞格
2. 当禁飞区为三格直线时，规划结果不包含禁飞格
3. 若某些特殊起飞位置与禁飞区组合导致不可行，应返回明确错误

## 风险与取舍

### 风险 1：Qt 到 Python 的调用边界

这是本轮最大的工程风险。当前代码库天然是“Python 生产消息、Qt 负责显示”，而新需求要求“Qt 发起重规划”。  
因此需要决定是：

- 让 Qt 直接嵌入一次规划调用
- 还是新增一个本地规划服务接口

推荐优先选“本地轻量桥接”，避免把 Python 逻辑硬塞进 Qt UI 线程。

### 风险 2：状态混乱

如果不使用显式状态机，而是直接根据按钮文案、选中数量、若干布尔值拼接逻辑，后续很容易出现：

- 文案已经切换但地图还可乱点
- 生成失败后状态无法回退
- 候选禁飞区和正式禁飞区混在一起

因此显式状态机是必要的，不是锦上添花。

### 风险 3：比赛语义与演示语义不一致

如果做成“点满三格自动生成”，虽然操作快，但不利于给评委展示“禁飞区设置完成后，再明确生成航线”的过程。  
本轮保留显式 `航线生成` 按钮，是更符合比赛展示逻辑的取舍。

## 不在本轮范围内

本轮不做以下内容：

- 执行中动态修改禁飞区
- 暂停任务后重新规划
- 任意形状禁飞区编辑
- 鼠标拖拽框选
- 多套任务配置持久化管理
- 地面站替代 Python 模拟端成为完整任务源

## 结论

本轮应以 **“单按钮 + 地图三格点选 + 显式状态机 + 生成后统一刷新”** 为核心方案推进。

这是当前代码库中改动最小、比赛语义最清晰、测试边界最明确的实现路径。  
其中真正的实现重点不是路径算法，而是：

- UI 状态管理
- 三格直线合法性校验
- Qt 到 Python 的重规划桥接
- 生成成功后的统一配置同步
